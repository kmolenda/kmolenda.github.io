Typy w języku C# - skrócony przegląd

Krzysztof Molenda 2019-10-04 C#

Programowanie w języku C# polega na definiowaniu i wykorzystaniu typów danych. Z formalnego punktu widzenia program w C# (również w Java) jest zbiorem deklaracji typów.

Dla programów wykonywalnych jeden z deklarowanych typów jest klasą zawierającą metodę Main().

Typ danych jest swego rodzaju szablonem, na bazie którego tworzona jest struktura danych (nie jest strukturą danych, a jedynie określa charakterystykę obiektów tworzonych na bazie takiego szablonu). Typ danych określony jest przez nazwę, wewnętrzną reprezentację danych (później wypełnianą właściwymi danymi) oraz właściwości/funkcjonalności i ograniczenia z nimi związane.

Instancjonowanie typu to tworzenie obiektu (bytu programistycznego) na bazie typu. Taki byt nazywany jest obiektem lub ogólniej - instancją. Każda dana w programie C# jest instancją pewnego typu - dostarczonego na poziomie języka, dostarczonego w bibliotekach standardowych (BCL) lub innych dołączonych bibliotekach, albo zdefiniowanego przez użytkownika.

Typami zdefiniowanymi na poziomie języka są np. typy liczbowe (int, double, ...), tablice (int[], int[][], double[,], ...), napisy (string).
Typami dostarczonymi w BCL są np. lista (System.Collections.Generic.List<T>) - generalnie wszystko dołączane klauzulą using na początku kodu programu.

Typy dzielimy na kategorie:

  • typy wartościowe (zwane czasami bezpośrednimi) - ang. value types
  • typy referencyjne (zwane czasami obiektowymi) - ang. reference types
  • typy wskaźnikowe - ang. pointer types (nie omawiane teraz)

Z technicznego punktu widzenia, instancje typów wartościowych z reguły przechowywane są na stosie (stack). Z semantycznego punktu widzenia, typy wartościowe opisują wartości, a te zawsze kopiowane (powielane) są przez wartość (operator przypisania).

struct Point
{
  public int X {get; set;}
  public int Y {get; set;}
}
var a = new Point(1,2); // Point jest strukturą
var b = a; // b jest niezależną kopią a

Dla instancji typów referencyjnych, z technicznego punktu widzenia dane przechowywane są na stercie (heap). Z punktu widzenia semantyki zmienne typów referencyjnych wskazują na .... Kopia zmiennej referencyjnej wskazuje na ten sam obiekt.

class Point
{
  public int X {get; set;}
  public int Y {get; set;}
}
var a = new Point(1,2); // Point jest klasą
var b = a; // b wskazuje na ten sam obiekc, na który wskazuje a

Architektura typów języka C#

Klasy "specjalne"

Obserwując powyższą hierarchię klas, zwróćmy uwagę na klasy "specjalne":

  • abstract public class ValueType - klasa osadzona w konstrukcjach języka C#. Wywodzą się z niej wszystkie typy wartościowe (instancje tych typów przechowywane są na stosie). Z klasy ValueType nie można bezpośrednio tworzyć instancji (bo jest abstrakcyjna). Używając słowa kluczowego struct do deklaracji struktury, formalnie tworzymy związek dziedziczenia między naszą tworzoną strukturą (np. MojaStruktura) a klasą ValueType.

  • abstract public class Enum - klasa dziedzicząca z ValueType. Nie można z niej bezpośrednio tworzyć instancji. "Obsługuje" słowo kluczowe enum. Definiując własne typy wyliczeniowe za pomocą słowa kluczowego enum formalnie budujemy związek dziedziczenia między naszym wyliczeniem (np. MojEnum) a klasą Enum.

  • abstract public class Array - klasa bazowa dla konstrukcji tablic w C#, dostarcza również metody do tworzenia, manipulowania, wyszukiwania i sortowania tablic. Formalnie, wszystkie tablice użytkownika (np. int[ ], double[ , ], int[][], ...) są typami pochodnymi z tej klasy.

Klasy statyczne

Klasy statyczne - w zasadzie nie różnią się niczym w budowie od klas "zwykłych", z jednym drobnym (a może poważnym) wyjątkiem - składają się wyłącznie ze składników statycznych (statycznych pól, statycznych metod i właściwości, statycznych konstruktorów). Niesie to konsekwencje niemożliwości utworzenia instancji tych klas (zatem nie można użyć operatora new):

// klasa System.Math jest statyczna
var math = new Math();

  error CS0723: Nie można zadeklarować zmiennej typu statycznego "Math"
  error CS0712: Nie można utworzyć wystąpienia klasy statycznej "Math"

Klasy statyczne, na poziomie wygenerowanego kodu IL oznaczone są jako sealed i abstract. Zatem, z klas statycznych nie można również dziedziczyć!

Składniki klasy statycznej są statyczne, a więc są osiągalne poprzez podanie nazwy klasy, a następnie nazwy składnika:

// klasa System.Math jest statyczna
var x = Math.Min(2, 3);
Console.WriteLine(x);

Klasy statyczne wykorzystywane są jako kontenery (biblioteki) dla zbioru metod o charakterze użytkowym. Metody takie otrzymują wymagane parametry i wykonują określone działania lub obliczenia, nie zmieniając stanu pól klasy, w której są umieszczone. Przykładem jest klasa System.Math zawierająca stałe związane z matematyką (liczby E, PI) oraz funkcje matematyczne (Sin, Sqrt, ...). Dawniej, w programowaniu proceduralnym i modularnym, tego typu konstrukcję nazywano modułem.

Typy anonimowe

Jak sama nazwa wskazuje, typ anonimowy to typ, którego programista nie nazwał. Nie nazwał, bo nie było takiej potrzeby - ponieważ był potrzebny tylko w celu wytworzenia obiektu tymczasowego (chwilowego).

Ponieważ nie wiadomo, jak typ się nazywa, więc jego deklaracja odbywa się z zastosowaniem słowa kluczowego var i inicjalizatora obiektu.

var x = new
            {
                imie = "Molenda",
                wiek = 55,
                pensja = 4000.00,
                zatrudniony = true,
            };
Console.WriteLine(x);
Console.WriteLine(x.GetType().ToString());

Uwaga: składniki typu anonimowego wnioskowane są na podstawie typów przypisanych wartości. Realizowany jest jako obiekt z auto-properties typu get (czyli tylko do odczytu). Zatem nie można zmienić wartości obiektu typu anonimowego. Typ anonimowy jest typem referencyjnym. Typ anonimowy nie zawiera metod definiowanych przez użytkownika.

Typ anonimowy (jako służący do tworzenia zmiennych tymczasowych) będzie zawsze lokalny dla metod, w których jest definiowany. W takim razie nie możesz przekazywać zmiennych typu anonimowych do innych metod (za wyjątkiem typu dynamic).

Zmienne typów anonimowych powszechnie wykorzystystywane są do budowania wyrażeń funkcyjnych (lambda) oraz w technologii LINQ.

Typy dynamiczne

Język C# językiem statycznego typowania - w trakcie tworzenia kodu muszą być znane typy zmiennych. Typ dynamiczny wprowadzono w C# 4 po to, aby odroczyć weryfikację typu na fazę uruchomienia. Okazało się to koniecznym w sytuacji integracji kodu C# z bibliotekami kodu skompilowanymi dla innych języków (głównie niezarządzalnych). W praktyce, wytwarzając kod wyłącznie w C# nie stosuje się typów dynamicznych.

  dynamic x = 112;
  Console.WriteLine($"wartosc: {x}, Typ: {x.GetType()}");

  x = "to jest tekst";
  Console.WriteLine($"wartosc: {x}, Typ: {x.GetType()}");

  x = true;
  Console.WriteLine($"wartosc: {x}, Typ: {x.GetType()}");

  x = DateTime.Now;
  Console.WriteLine($"wartosc: {x}, Typ: {x.GetType()}");

Ponieważ kompilator omija deklaracje typu anonimowego (nie wykonuje weryfikacji poprawności typów), zatem ewentualne błędy nie zostaną zgłoszone podczas tworzenia kodu i kompilacji. Ujawniają się dopiero podczas uruchomienia.

VisualStudio nie wspiera IntelliSense dla typów dynamicznych.

Typy krotki. Tuple, ValueTuple

W praktyce programistycznej często potrzebujemy struktury danych reprezentowanej jako ciąg wartości potencjalnie różnych typów (gdyby typ był ten sam - mamy tablicę bądź listę).

Od C# 4 BCL dostarcza predefiniowanego generycznego typu Tuple. Jest on zadeklarowany w 8 wariantach: Tuple<T1>, Tuple<T1, T2>, ..., Tuple<T1, ..., T8, TRest>. Dostęp do składników jest możliwy za pomocą właściwości Item1, ..., Item7, Rest. Od C# 7 mamy możliwość dostępu poprzez zdefiniowane nazwy składników (dekonstrukcję).

var osoba = Tuple.Create("Molenda", 55, 4000.0, true); // C# 4
Console.WriteLine( $"nazwisko = {osoba.Item1}, wiek = {osoba.Item2}");

var osoba1 = ("Molenda", 55, 4000.0, true); // C# 7, ValueTuple
Console.WriteLine( $"nazwisko = {osoba1.Item1}, wiek = {osoba1.Item2}");

// C# 7 Value named tuple
var osoba2 = (nazwisko: "Molenda", wiek: 55, pensja: 4000.0, zatrudniony: true); // C# 7
Console.WriteLine( $"nazwisko = {osoba2.nazwisko}, wiek = {osoba2.wiek}");

(int min, int max) MinMax(params int[] liczby)
{
  int minimum, maximum;
  minimum = maximum = liczby[0];
  foreach( var x in liczby)
  {
    if( x < minimum ) minimum = x;
    if( x > maximum ) maximum = x;
  }
  return (minimum, maximum);
}

(int x, int y) = MinMax(2, 1, 6, 3, 1);
Console.WriteLine( $"minimum = {x}, maximum = {y}" );

// value tuple deconstrucion
var (a, b) = MinMax(2, 1, 6, 3, 1);
Console.WriteLine( $"minimum = {a}, maximum = {b}" );

var (min, _) = MinMax(2, 1, 6, 3, 1); // ignorujemy zmienne niepotrzebne
Console.WriteLine( $"minimum = {min}" );

Cechy typów

Wartość domyślna

Wartość domyślną pobierzemy poleceniem default(type) lub dla C# 7 po prostu default.

Wartość domyślna:

  • dla typów wartościowych liczbowych: 0
  • dla typu wartościowego bool: false
  • dla typu wartościowego char: \0
  • dla typów referencyjnych: null

Dla typów użytkownika:

  • dla struct wartości domyślne pól struktury
  • dla class zawsze null.

Pewną formą określania wartości domyślnej dla typów referencyjnych jest definiowanie konstruktora domyślnego (bezargumentowego). Np. dla ułamka możemy przyjąć 0/1

public class Ulamek
{
  public int Licznik {get; private set;}
  private int mianownik;
  public int Mianownik
  {
    get { return mianownik;}
    private set
    {
      if (value == 0) throw new DivideByZeroException();
      mianownik = value;
    }
  }

  public Ulamek() { Licznik = 0; Mianownik = 1; }
  public override string ToString() => $"{Licznik}/{Mianownik}";
}

Konwersje

Casting and type conversions (C# Programming Guide)

Konwersje domyślne (implicit)

Konwersje domyślne (ang. implicit) wykonywane są automatycznie wtedy, kiedy kontekst instrukcji/wyrażenia na to wskazuje. Ogólna zasada: typ węższy jest konwertowany na typ szerszy (pojemniejszy), o ile nie nastąpi utrata informacji.

int x = 2;
double y = x; // OK. y = 2
long z = y; // błąd kompilacji

error CS0266: Nie można niejawnie przekonwertować typu "double" na "long". Istnieje konwersja jawna (czy nie brakuje rzutu?).

Projektując typ można definiować konwersje niejawne, przeciążając operator konwersji:

public class Ulamek
{
  public int Licznik {get; private set;}
  private int mianownik;
  public int Mianownik
  {
    get { return mianownik;}
    private set
    {
      if (value == 0) throw new DivideByZeroException();
      mianownik = value;
    }
  }

  public Ulamek() { Licznik = 0; Mianownik = 1; }
  public Ulamek(int licznik, int mianownik) { Licznik = licznik; Mianownik = mianownik; }
  public override string ToString() => $"{Licznik}/{Mianownik}";

  // konwersja domyślna z `int` na `Ulamek`
  public static implicit operator Ulamek(int value) => new Ulamek(value, 1);
}

// ...
Ulamek u = 5;
Console.WriteLine( u );

public class Ulamek
{
  public int Licznik {get; private set;}
  private int mianownik;
  public int Mianownik
  {
    get { return mianownik;}
    private set
    {
      if (value == 0) throw new DivideByZeroException();
      mianownik = value;
    }
  }

  public Ulamek() { Licznik = 0; Mianownik = 1; }
  public Ulamek(int licznik, int mianownik) { Licznik = licznik; Mianownik = mianownik; }
  public override string ToString() => $"{Licznik}/{Mianownik}";

  // konwersja domyślna z `int` na `Ulamek`
  public static explicit operator int(Ulamek u) => u.Licznik / u.Mianownik;
}

// ...

Opakowanie (boxing), odpakowanie (unboxing)

Boxing and Unboxing (C# Programming Guide)

Kurs: Obiektowość w C# : System.Object boxing i unboxing, rzutowanie : 06

  • Boxing - proces konwertowania typu wartościowego na typ referencyjny. Proces ten wykonywany jest automatycznie:

    int i = 10;
object o = i; //boxing

  • Unboxing - proces konwertowania typu referencyjnego na wartościowy (proces odwrotny do boxing). Wywoływany jest przez jawne rzutowanie

    object o = 10;  // `o` przechowuje wartość typu `int`
int i = (int)o; //unboxing

Konwersje boxing/unboxing wykonują kopie wartości, zatem zmiana wartości oryginału nie wpływa na wartość po konwersji.

int i = 10;
object o = i; // boxing
o = 20;
Console.WriteLine(i); // output: 10

Przykład błędnej konwersji

int i = 10;
object o = i; // boxing
double d = (double)o; // runtime exception

Poprawna wersja:

int i = 10;
object o = i; // boxing
double d = (double)(int)o; // poprawnie

Opakowywanie i odpakowywanie pochłania czas, zatem należy unikać jego stosowania.
