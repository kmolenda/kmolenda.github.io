W C# typy dzielimy na kategorie:

  • typy wartościowe (zwane czasami bezpośrednimi) - ang. value types
  • typy referencyjne (zwane czasami obiektowymi) - ang. reference types
  • typy wskaźnikowe - ang. pointer types

Z technicznego punktu widzenia, instancje typów wartościowych z reguły przechowywane są na stosie (stack). Z semantycznego punktu widzenia, typy wartościowe opisują wartości, a te zawsze kopiowane (powielane) są przez wartość (operator przypisania).

struct Point
{
  public int X {get; set;}
  public int Y {get; set;}

  public Point(int x, int y) {X = x; Y = y;}
}
var a = new Point(1,2); // Point jest strukturą
var b = a; // b jest niezależną kopią a

    Typy wartościowe użytkownika - struct

    Użytkownik może definiować własne typy wartościowe za pomocą słowa kluczowego struct. Maja one składniki: pola, właściwości, metody -- podobnie jak klasy. Składnia jest również podobna do tej w klasach.

    • Klasy są typami referencyjnymi, struktury typami wartościowymi
    • Typy strukturalne formalnie, dziedziczą z klasy abstrakcyjnej ValueType
    • Struktury są domyślnie zapieczętowane (sealed), zatem nie można z nich dziedziczyć
    • Zmienne typów strukturalnych nie mogą być null (wartości nullables później ...)
    • Dwie zmienne strukturalne nie mogą odwoływać się do tego samego obiektu

  public int Y {get; set;}

    Konstruktor, inicjalizacja

    • Nie można inicjalizować pól i właściwości struktury

      struct Time
{
    private int hours = 0; //compile error
    private int minutes; // inicjalizacja na 0 ( default(int) )
    private int seconds; // inicjalizacja na 0 ( default(int) )
}

      error CS0573: '"Time": nie można umieścić inicjatorów właściwości lub pola wystąpienia w strukturach

    • Nie można deklarować konstruktora domyślnego (bezparametrowego). Kompilator zawsze go generuje. Przypisuje on wartości domyślne typów do pól/właściwości. Konstruktora domyślnego nie można ukryć / usunąć / przesłonić / przedefiniować ...

      struct Time
{
    private int hours, minutes, seconds;

    public int Hours => hours;
    public int Minutes => minutes;
    public int Seconds => seconds;

    public Time() //compile error: default constructor not allowed
    {
      hours = minutes = seconds = 0;
    }
}

      error CS0568: Struktury nie mogą zawierać jawnych konstruktorów bez parametrów

    • Aby przypisać niedomyślne wartości do pól/właściwości struktury, można utworzyć konstruktor z parametrami.

      ⚠️ W konstruktorze należy przypisać wartości do wszystkich pól.

      struct Time
{
    private int hours, minutes, seconds;

    public int Hours => hours;
    public int Minutes => minutes;
    public int Seconds => seconds;

    public Time(int hh, int mm)
    {
      hours = hh;
      minutes = mm;
    } //compile error: seconds not initialized
}

      warning CS0649: Do pola "Time.seconds" nigdy nie jest przypisywana wartość i będzie ono mieć zawsze wartość domyślną 0
      error CS0171: Pole "Time.seconds" musi być całkowicie przypisane przed zwróceniem sterowania do elementu wywołującego

      Ponieważ konstruktor musi jawnie inicjalizować wszystkie pola, aby wprowadzić przeciążone wersje konstrukcji instancji Time, używamy w konstruktorze parametrów opcjonalnych:

      struct Time
{
    private int hours, minutes, seconds; // inicjalizacja automatyczna na default(int), czyli 0

    public int Hours => hours; // get, read-only
    public int Minutes => minutes; // get, read-only
    public int Seconds => seconds; // get, read-only

    public Time(int hours, int minutes = 0, int seconds = 0)
    {
            this.hours = verify(hours, 0, 23);
            this.minutes = verify(minutes, 0, 59);
            this.seconds = verify(seconds, 0, 59);

        // funkcja lokalna, C# 7
        int verify(int value, int min, int max) 
            => (value >=min && value <=max )? value : throw new ArgumentOutOfRangeException();
    } // koniec konstruktora

    public override string ToString() => $"{Hours}:{Minutes:D2}:{Seconds:D2}";
}

// Main()
Time t1 = new Time(); //konstruktor domyślny, bezparametrowy
Console.WriteLine( t1 );

Time t2; // deklaracja, żaden kostruktor nie jest uruchamiany
// Console.WriteLine( t2 ); //error CS0165: Użyto nieprzypisanej zmiennej lokalnej "t2"

Time t3 = new Time(13); // konstruktor własny
Console.WriteLine( t3 );

Time t4 = new Time(minutes: 13, hours: 8); // konstruktor własny
Console.WriteLine( t4 );

    • Można utworzyć instancję struktury bez jawnego użycia operatora new, ale pod pewnymi warunkami:

      • nie możesz użyć wartości pól struktury, dopóki ich jawnie nie zainicjujesz,
      • nie możesz użyć żadnej metody struktury, dopóki wszystkie składniki nie zostaną jawnie zainicjowane.

      Przykład:

      struct Wektor
{
  public double X;
  public double Y;

  public double Dlugosc => Math.Sqrt( X*X + Y*Y );
  public override string ToString() => $"[{X}, {Y}]";
}
      ▶︎
      all
      running...

      W poniższym kodzie tworzymy zmienną strukturalną w i przypisujemy jednemu z jej pól wartość. Z tej wartości możemy korzystać (ale tylko z tej wartości).

      Wektor w;
w.X = 1;
Console.WriteLine( w.X ); // output: 1
Console.WriteLine( w.Dlugosc ); // błąd kompilacji 
                                // (próba odczytania zmiennej niezainicjowanej)

      Jeśli zainicjujemy oba pola, możemy w pełni korzystać z metod struktury:

      Wektor w;
w.X = 1;
w.Y = 2;
Console.WriteLine( w );
// output: [1, 2]

      Problem pojawia się, jeśli zamiast pól zastosujemy properties:

      struct Wektor
{
  public double X {get; set;}
  public double Y {get; set;}

  public double Dlugosc => Math.Sqrt( X*X + Y*Y );
  public override string ToString() => $"[{X}, {Y}]";
}
      ▶︎
      all
      running...

      W tym przypadku, w sytuacji użycia instancji struktury, kompilator stwierdzi, że zmienna nie jest zainicjowana:

      Wektor w; // compilation error: próba użycia zmiennej nie zainicjowanej
w.X = 1;
w.Y = 2;
Console.WriteLine( w );

      Sytuacja ta występuje nawet wtedy, kiedy inicjujemy properties. Dlaczego?

      Properties są - z formalnego punktu widzenia - metodami. W tym przypadku użyliśmy auto-properties. Kompilator wygenerował pola anonimowe (dla nas), które "przykrył" właściwościami X oraz Y. I te pola nie zostały zainicjowane, zaś my próbujemy skorzystać z metody (czyli property X).

      Rozwiązaniem jest jawne użycie konstruktora: Wektor w = new Wektor(); (wtedy składniki otrzymają wartości domyślne), a następnie zmiana wartości tych składników.

      Innym sposobem jest użycie inicjalizatora Wektor w = new Wektor {X = 1, Y = 1}; (o ile properties są read-write i publiczne).

    • Jeśli struktura zawiera pole typu referencyjnego, to dla niego trzeba wywołać konstruktor tego typu referencyjnego jawnie:

      public struct Student
{
    public string Nazwisko { get; set; }
    public int RokStudiow { get; set; }
    public List<int> Oceny { get; }

    public Student(string nazwisko, int rokStudiow)
    {
        Nazwisko = nazwisko;
        RokStudiow = rokStudiow;
        Oceny = new List<int>();
    }

    public override string ToString() 
      => $"Student {Nazwisko}, rok {RokStudiow}, oceny: {String.Join(',', Oceny)}";
}

      ⚠️ Należy unikać struktur z polami typu referencyjnego (wyjaśnienie powodu później). W powyższym przypadku lepszym rozwiązaniem byłoby użycie klasy, a nie struktury.

      Przykład kodu wykorzystującego strukturę Student:

      Student s = new Student("Molenda", 2);
s.Oceny.Add(2);
s.Oceny.AddRange(new int[] {2, 3, 4});
Console.WriteLine(s);
// Student Molenda, rok 2, oceny: 2,2,3,4

    • Struktury mogą zawierać składniki statyczne - stałe, pola statyczne i metody statyczne. Do ich inicjalizacji posłuży konstruktor statyczny, deklarowany jawnie. Zasady jego użycia i działania są podobne jak dla statycznego konstruktora dla klasy (może być tylko jeden, jest bezparametrowy, nie może odwoływać się do składników instancji, nie podajemy modyfikatora dostępu - jest private). Są jednak pewne subtelne różnice - jest uruchamiany jednorazowo, automatycznie, wyzwalany przez wcześniej pojawiające się zdarzenie:

      • wywołanie dowolnego statycznego składnika instancji,
      • jawne wywołanie zadeklarowanego konstruktora.

      A static constructor is used to initialize any static data, or to perform a particular action that needs to be performed once only. It is called automatically before the first instance is created or any static members are referenced.

      • A static constructor cannot be called directly.
      • The user has no control on when the static constructor is executed in the program.
      public struct Test
{
    public const int Max = 10;
    public int Value;
    public static double Ratio;
    static Test() // static constructor
    {
        Ratio = 0.5;
        Console.WriteLine($"static constructor, ratio = {Ratio}");
    }

    public Test(int value) // instance parametrized constructor
    {
        Value = value;
        Console.WriteLine("instance constructor");
    }
}

// === Main() ===
Test test;                       // output:
test.Value = 5;                  // output:
Console.WriteLine( test.Value ); // output: 5
Console.WriteLine( Test.Max );   // output: 10
Test.Ratio = 1;                  // output: static constructor, ratio = 0,5
Console.WriteLine( Test.Ratio ); // output: 1

Test test2 = new Test(6);        // output: instance constructor

      Pola statyczne mogą być inicjalizowane bezpośrednio w deklaracji.

    • Destruktor nie może wystąpić w deklaracji struktury. Garbage Collector nie zarządza pamięcią stosu, nie pracuje więc z typami strukturalnymi.

    Podsumowanie - konstruktor, inicjalizacja

    • Konstruktor domyślny struktury zawsze jest dostępny. Inicjalizuje pola struktury wartościami domyślnymi typów pól.

    • Konstruktory użytkownika są opcjonalne, ale jeśli są, to muszą zawierać parametry.

    • Pola nie mogą być inicjalizowane w ciele struktury (w deklaracjach), chyba, że są statyczne. Mogą być inicjalizowane w konstruktorach lub poza strukturą, po jej deklaracji - o ile są to pola publiczne read write.

    • Pola prywatne mogą być zainicjowane wyłącznie w konstruktorze.

    • Utworzenie zmiennej strukturalnej bez jawnego wywołania konstruktora (operatorem new) nie powoduje, że pola struktury będą zainicjowane.

    • Dopóki pole nie zostanie zainicjowane, nie można go użyć. Dopóki wszystkie pola struktury nie zostaną zainicjowane, nie można użyć żadnej metody zdefiniowanej w strukturze (również properties).

    • Jeśli struktura zawiera typy referencyjne (klasy) jako swoje składniki, to muszą one zostać jawnie zainicjowane ich konstruktorami. Należy unikać definiowania struktur ze składnikami referencyjnymi.

    • Konstruktor statyczny inicjalizuje składniki statyczne (jak w klasach), jest uruchamiany jednorazowo (jak w klasach), ale bezpośrednio przed pierwszym użyciem dowolnego składnika statycznego (inaczej niż w klasach, gdzie uruchamiany jest przed utworzeniem instancji klasy lub odwołaniem do składnika statycznego).

    • Destruktor nie jest dozwolony dla struktur.

    Struktury i dziedziczenie

    Wszystkie typy strukturalne dziedziczą ze specjalnej, abstrakcyjnej klasy System.ValueType. Specjalność tej klasy polega na silnym związku z gramatyką języka. Dziedziczenie z tej klasy realizowane jest za pomocą słowa kluczowego struct - zatem wszystkie struktury z niej pochodzą.

    • Typy wartościowe są domyślnie sealed - zapieczętowane. Zatem łańcuch dziedziczenia dla struktur zaczyna się od klasy object, przechodzi przez System.ValueType i kończy się na implementowanej strukturze. Struktury nie biorą udziału w dziedziczeniu - nie mogą dziedziczyć z innych struktur czy klas. Mogą jednak implementować interfejsy.

    • Skoro struktury nie biorą udziału w dziedziczeniu, zatem niektóre z modyfikatorów dostępu nie mają sensu: abstract, protected, protected internal, virtual czy private protected wprowadzony w C# 7.2.

    • Jedynymi możliwymi słowami kluczowymi związanymi z dziedziczeniem, które można użyć przy definiowaniu struktury, są new oraz override. Wykorzystujemy je do definiowania składników struktury, które mają taką samą sygnaturę jak te zdefiniowane w klasie bazowej System.ValueType, w celu ich ukrycia lub przesłonięcia.

    • Struktury mogą implementować interfejsy. Wiele z nich, np. te które budują BCL (Base Class Library), implementuje takie interfejsy, jak np. IComparable, IEquatable czy IFormatable.

    Struktury i równość

    ⚠️ Struktury domyślnie nie implementują operatorów == oraz != -- w przeciwieństwie do klas, gdzie domyślnie oznaczają one równość/nierówność referencji -- o ile nie zostały przeciążone.

    Dla wcześniejszego przykładu ze strukturą Time:

    Time t1 = new Time(12, 00);
Time t2 = t1;
Console.WriteLine( $"t1 = {t1}, t2 = {t2}" ); // t1 = 12:00:00, t2 = 12:00:00
Console.WriteLine( t1 == t2 ); // błąd kompilacji
// error CS0019: Nie można zastosować operatora "==" 
// do argumentów operacji typu "Time" lub "Time"

    W klasie System.ValueType przesłonięta została natomiast metoda instancji public override bool Equals(object obj):

    Console.WriteLine( t1.Equals(t2) ); // true

    Zwraca ona true, jeśli instancje 1) są tego samego typu i 2) reprezentują tę samą wartość, zaś false w przeciwnym przypadku.

    Z tej implementacji korzysta statyczny wariant Equals:

    Console.WriteLine( object.Equals(t1, t2) );    // true
Console.WriteLine( ValueType.Equals(t1, t2) ); // true

    Wywołanie dla struktury metody statycznej object.ReferenceEquals zawsze zwraca false - co jest oczywiste. Ponieważ ReferenceEquals oczekuje parametrów, które są typami referencyjnymi, opakowuje te strukturalne do obiektów (boxing) i - mimo, iż obiekty te zawierają takie same wartości - są różnymi obiektami i mają różne referencje, więc zwraca false.

    Warto chwilę zastanowić się nad tym, jak zaimplementowano Equals w klasie System.ValueType? Oto fragment oficjalnej dokumentacji tej metody:

    ValueType.Equals(Object) Method
    public override bool Equals (object obj);

    • Namespace: System
    • Assemblies: System.Runtime.dll, mscorlib.dll, netstandard.dll
      Indicates whether this instance and a specified object are equal.
    • Parameters: obj Object
      The object to compare with the current instance.
    • Returns: Boolean
      true if obj and this instance are the same type and represent the same value; otherwise, false.

    Porównywanie odbywa się pole po polu (field-by-field). Jeśli wszystkie i odpowiednie pola są równe, instancje porównywanych struktur są również równe.

    Tylko jak to technicznie jest realizowane?

    For structs, the default implementation of Object.Equals(Object) (which is the overridden version in System.ValueType) performs a value equality check by using reflection to compare the values of every field in the type. When an implementer overrides the virtual Equals method in a struct, the purpose is to provide a more efficient means of performing the value equality check and optionally to base the comparison on some subset of the struct's field or properties. Źródło.
    If none of the fields of the current instance and obj are reference types, the Equals method performs a byte-by-byte comparison of the two objects in memory. Otherwise, it uses reflection to compare the corresponding fields of obj and this instance.

    Implementacja Equals w klasie ValueType https://referencesource.microsoft.com/#mscorlib/system/valuetype.cs:

    public abstract class ValueType {

[System.Security.SecuritySafeCritical]
public override bool Equals (Object obj) {
    BCLDebug.Perf(false, "ValueType::Equals is not fast.  "+this.GetType().FullName+" should override Equals(Object)");
    if (null==obj) {
        return false;
    }
    RuntimeType thisType = (RuntimeType)this.GetType();
    RuntimeType thatType = (RuntimeType)obj.GetType();

    if (thatType!=thisType) {
        return false;
    }

    Object thisObj = (Object)this;
    Object thisResult, thatResult;

    // if there are no GC references in this object we can avoid reflection 
    // and do a fast memcmp
    if (CanCompareBits(this))
        return FastEqualsCheck(thisObj, obj);

    FieldInfo[] thisFields = thisType.GetFields(BindingFlags.Instance | BindingFlags.Public | BindingFlags.NonPublic);

    for (int i=0; i<thisFields.Length; i++) {
        thisResult = ((RtFieldInfo)thisFields[i]).UnsafeGetValue(thisObj);
        thatResult = ((RtFieldInfo)thisFields[i]).UnsafeGetValue(obj);

        if (thisResult == null) {
            if (thatResult != null)
                return false;
        }
        else
        if (!thisResult.Equals(thatResult)) {
            return false;
        }
    }

    return true;
}
//...
}

    ⚠️ Jak wynika z powyższych cytatów porównywanie domyślne może nie być efektywne wydajnościowo i obowiązkiem implementatora struktury jest przesłonięcie tej metody, dostarczając własny -- i najprawdopodobniej wydajniejszy -- sposób porównywania.

    W artykule C# Journey into struct equality comparison, deep dive przedstawiono wyniki testów wydajnościowych, wskazujące na nieefektywność domyślnego porównywania dla struktur.

    Co należy zatem zrobić definiując poprawnie i kompletnie strukturę?

    1. Przesłoń metodę public override bool Equals(object obj).
    2. Nie zapomnij o przesłonięciu GetHashCode().
    3. Zaimplementuj interfejs IEquatable<T> dostarczając public bool Equals<T>(T other).
    4. Zaimplementuj przeciążoną metodę public static bool Equals<T>(T t1, T t2).
    5. Zdefiniuj przeciążenie operatorów == oraz !=.

    Oczywiście logikę porównywania powinieneś zawrzeć w jednej metodzie, zaś w pozostałych -- które w zasadzie udostępniają tę samą funkcjonalność w trochę inny sposób -- wywołać.

    Przykład:

    Dla wcześniej zdefiiniowanego typu Time dodajemy implementacje równości:

    // Equals implementation =============================================

public override bool Equals(object o)
{
  Debug.Write("this.Equals(object) -> ");
  if( o is null ) return false;
  if( o is Time t ) // C# 7
      return Hours == t.Hours && Minutes == t.Minutes && Seconds == t.Seconds;
  else
      return false;
}

public override int GetHashCode() => (hours, minutes, seconds).GetHashCode();

public static new bool Equals(object o1, object o2)
{
  Debug.Write("Object.Equals(object, object) -> ");
  if( (o1,o2) == (null,null) ) return true;
  if( (o1 is null) && !(o2 is null) ) return false;

  return o1.Equals(o2);
}

public bool Equals(Time other) //=> this.Equals((object)other); 
{
  Debug.Write("this.Equals(Time) -> ");
  return this.Equals((object)other); 
}
// uwaga: ... => this.Equals(other); daje rekurencyjne odwołanie.

public static bool Equals(Time t1, Time t2) //=> t1.Equals(t2);

    Struktury i readonly

    Typ strukturalny, realizujący koncepcję wartości, idealnie nadaje się do opisu typów niezmienniczych immutable (o wartościach myślimy w kategorii ich niezmienniczości, przypisanie jest faktycznie kopiowaniem wartości). Niektórzy wręcz stwierdzaja, że struktur mutables nie powinno się tworzyć (patrz: wątek na StackOverflow).

    Obiekty, które są niezmiennicze, są bezpieczniejsze w wykorzystaniu ich w programowaniu wielowątkowym.

    Immutability możemy zapewnić przemyślanym tworzeniem kodu (np. tylko gettersy, dbanie o niezmienianie wartości przez metody typu). Język C# wspiera poprawne tworzenie typów niezmienniczych poprzez odpowiednie wykorzystanie słowa kluczowego readonly.

    Pola oznaczone jako readonly: przypisanie możliwe jest tylko w części deklaracyjnej lub konstruktorze. Poza konstruktorem przypisanie nie jest możliwe.

    • Pola wartościowe readonly sa w pełni niezmiennicze.
    • Pola referencyjne readonly - niezmiennicza jest referencja (nie można podmienić obiektu), ale za jej pomocą można modyfikować zawartość obiektu (o ile on na to pozwala).

    Przykład:

    public struct Student
{
    public readonly string Nazwisko;
    public readonly int RokStudiow;
    public readonly List<int> Oceny;

    public Student(string nazwisko, int rokStudiow)
    {
        Nazwisko = nazwisko;
        RokStudiow = rokStudiow;
        Oceny = new List<int>();
    }

    public override string ToString()
      => $"Student {Nazwisko}, rok {RokStudiow}, oceny: {String.Join(',', Oceny)}";
}

    all
    running...

    1. Wszystkie składniki struktury są readonly, można przypisać im wartości jedynie w konstruktorze.

    2. Referencja Oceny wskazuje na listę ocen. Listy nie można zmienić (jest ona tworzona w konstruktorze, początkowo pusta), ale można nią zarządzać (wprowadzać i usuwać oceny).

      Student s = new Student("Molenda", 2);
s.Oceny.Add(2);

    Od C# 7.2 możliwe jest użycie readonly struct, w deklaracji struktury:

  // ...

    Jest to wymuszenie, aby struktura była niezmiennicza. Wszystkie jej pola domyślnie traktowane są jako readonly (nie ma obowiązku deklarowania pól jako readonly, ale nie jest to błędem).

    Struktury jako parametry metod i zwracane wartości

    Parametry metod

    • Struktury domyślnie przekazywane są do metod przez wartość -- tworzone są ich kopie i one wykorzystywane są w kodzie metody.

      public struct Time { ... }

// intencja - zamiana miejscami t1 ⇆ t2
public void Swap(Time t1, Time t2)
{
  Time temp = t1;
  t1 = t2;
  t2 = temp;
}

Time t1 = new Time(12, 0, 0);
Time t2 = new Time(13, 30, 0);
Swap(t1, t2);
Console.WriteLine($"t1 = {t1}, t2 = {t2}");
// output: t1 = 12:00:00, t2 = 13:30:00
// nie działa

    • Zmianę domyślnego przekazywania parametrów wymusimy słowami kluczowymi ref, out. W tym przypadku przekazywane są referencje do struktur, zatem możliwa jest zmiana danych. Nie jest wykonywany boxing.

      public struct Time { ... }

// zamiana miejscami t1 ⇆ t2
public void Swap(ref Time t1, ref Time t2)
{
  Time temp = t1;
  t1 = t2;
  t2 = temp;
}

Time t1 = new Time(12, 0, 0);
Time t2 = new Time(13, 30, 0);
Swap(ref t1, ref t2);
Console.WriteLine($"t1 = {t1}, t2 = {t2}");
// output: t1 = 13:30:00, t2 = 12:00:00

    • W C# 7.2 wprowadzono modyfikator in. Motywacją były aspekty wydajnościowe (https://docs.microsoft.com/en-us/dotnet/csharp/language-reference/proposals/csharp-7.2/readonly-ref#motivation). Parametr strukturalny z modyfikatorem in może być przekazany do metody przez referencję (a więc kopia nie jest wykonywana), za pomocą tej referencji mamy dostęp do składników, ale referencji tej wewnątrz metody nie wolno zmienić. Przekazanie referencji nie uruchamia boxingu.

      // public readonly struct Student { ... }

void DopiszOceny( in Student student, params int[] oceny )
{
  // dostęp do składników
  student.Oceny.AddRange(oceny);
  Console.WriteLine(student);
  //student = new Student("Abacki", 1); // błąd kompilacji
}

Student stud = new Student("Molenda", 1);
DopiszOceny( in stud, 2, 3, 4 );

      Uzycie in w sygnaturze metody jest tylko deklaracją możliwości przekazania parametru przez referencję. O faktycznym wykorzystaniu decyduje postać wywołania metody. Dla poprzedniego przykładu:

      • DopiszOceny( stud, 2, 3, 4) - przekazanie struktury stud przez kopię,
      • DopiszOceny( in stud, 2, 3, 4) - przekazanie struktury stud przez referencję poprzez jawne użycie in w wywołaniu metody.

      Za pomocą parametru in mozemy przekazywać parametry strukturalne, które są immutable lub mutable. W przypadku tych pierwszych, niezmiennicznych, uzyskujemy faktycznie zwiększenie wydajności (nie ma kopiowania, nie ma boxingu). Kłopot pojawia się, jeśli struktura zmienia swój stan w trakcie wykorzystania. Użycie parametru in powoduje pewne wymuszenia przy generowaniu kodu:

      1. Parametr poprzedzony in staje się readonly.
      2. Aby zadbać o powyższe, kompilator -- dla bezpieczeństwa -- przy wywołaniu musi stworzyć kopię wartości parametru, która będzie readonly. Na tej kopii działa metoda. Zatem trudno mówić o jakiejkolwiek optymalizacji.

      Więcej na ten temat: https://docs.microsoft.com/pl-pl/dotnet/csharp/write-safe-efficient-code#never-use-mutable-structs-as-in-in-argument

    Typ ref struct

    W C# 7.2 (z modyfikacjami w C# 8) wprowadzono możliwość deklarowania struktur z parametrem ref (np. public ref struct MyStruct). Motywacją było dostarczenie wyspecjalizowanego typu, który w niektórych sytuacjach dałby mozliwość tworzenia bardziej efektywnego kodu.

    Typy strukturalne w bibliotekach C#

    W bibliotekach C# zaimplementowano wiele typów strukturalnych.

    • Wszystkie typy liczbowe (int, long, float, ...) realizowane sa jako struktury (opowiednio System.Int32, System.Int64, System.Single, ...).

    • Typy bool (System.Boolean) oraz char (System.Character) są typami strukturalnymi.

    • Point czy Rectangle są strukturami zdefiniowanymi w System.Drawing.

    • BigInteger - wielka liczba całkowita, Complex - liczba zespolona - są strukturami zdefiniowanymi w System.Numerics.

    • DateTime oraz TimeSpan - reprezentacje daty i czasu oraz upływu czasu.

    • Nullable<T> jest strukurą opakowującą inny typ strukturalny T, dostarczając możliwość przypisania null.

    • Void - specjalna struktura określająca zwracany typ dla metod, które nie zwracają wartości (procedur) - głównie wykorzystywana w System.Reflection.

    Podsumowanie

    Structs share most of the same syntax as classes. The name of the struct must be a valid C# identifier name. Structs are more limited than classes in the following ways:

    • Within a struct declaration, fields cannot be initialized unless they are declared as const or static.
    • A struct cannot declare a parameterless constructor (a constructor without parameters) or a finalizer.
    • Structs are copied on assignment. When a struct is assigned to a new variable, all the data is copied, and any modification to the new copy does not change the data for the original copy. This is important to remember when working with collections of value types such as Dictionary<string, myStruct>.
    • Structs are value types, unlike classes, which are reference types.
    • Unlike classes, structs can be instantiated without using a new operator.
    • Structs can declare constructors that have parameters.
    • A struct cannot inherit from another struct or class, and it cannot be the base of a class. All structs inherit directly from ValueType, which inherits from Object.
    • A struct can implement interfaces.
    • A struct cannot be null, and a struct variable cannot be assigned null unless the variable is declared as a nullable value type.

    Guidelines:

    ✓ CONSIDER defining a struct instead of a class if instances of the type are small and commonly short-lived or are commonly embedded in other objects.

    X AVOID defining a struct unless the type has all of the following characteristics:

    • It logically represents a single value, similar to primitive types (int, double, etc.).
    • It has an instance size under 16 bytes.
    • It is immutable.
    • It will not have to be boxed frequently.

    In all other cases, you should define your types as classes.

    X DO NOT provide a parameterless constructor for a struct.

    Following this guideline allows arrays of structs to be created without having to run the constructor on each item of the array. Notice that C# does not allow structs to have parameterless constructors.

    X DO NOT define mutable value types.

    Mutable value types have several problems. For example, when a property getter returns a value type, the caller receives a copy. Because the copy is created implicitly, developers might not be aware that they are mutating the copy, and not the original value. Also, some languages (dynamic languages, in particular) have problems using mutable value types because even local variables, when dereferenced, cause a copy to be made.

    ✓ DO ensure that a state where all instance data is set to zero, false, or null (as appropriate) is valid.

    This prevents accidental creation of invalid instances when an array of the structs is created.

    ✓ DO implement IEquatable<T> on value types.

    The Object.Equals method on value types causes boxing, and its default implementation is not very efficient, because it uses reflection. Equals can have much better performance and can be implemented so that it will not cause boxing.

    X DO NOT explicitly extend ValueType. In fact, most languages prevent this.

    In general, structs can be very useful but should only be used for small, single, immutable values that will not be boxed frequently.
