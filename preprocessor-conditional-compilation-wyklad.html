Dyrektywy preprocesora, kompilacja warunkowa

Preprocesor C#

Preprocesor
program komputerowy, którego zadaniem jest przetworzenie kodu źródłowego, w sposób określony przez programistę za pomocą dyrektyw preprocesora, na kod wyjściowy. Tak przetworzony kod źródłowy poddawany jest następnie analizie składniowej, kompilacji, a w końcu konsolidacji.

Kompilator języka C# nie ma oddzielonego preprocesora - jest on z nim zintegrowany. Sterowanie preprocessingiem odbywa się za pomocą dyrektyw.

Głównym zastosowaniem preprocesora C# jest warunkowa kompilacja kodu. Nie dostarcza on (jak w C/C++) możliwości budowania makr.

Dyrektywy preprocesora C#

Dyrektywa
linia w kodzie, która nie jest częścią logiki programu. Dyrektywa dostarcza informacji o tym, jak kompilator ma zadziałać, a nie jak wykonywać się mają instrukcje, czy jak ma się zachowywać pamięć.

Dyrektywy preprocesora są wierszami w kodzie C# rozpoczynającymi się od znaku # po którym następuje nazwa dyrektywy. W danym wierszu może wystąpić tylko jedna dyrektywa preprocesora. Mogą wystąpić spacje przed oraz po znaku #.

Dyrektywy preprocesora C#, w podziale ze względu na przeznaczenie:

  • Warunkowa kompilacja - włączanie lub wyłączanie fragmentów kodu z kompilacji:

    #if
#else
#elif
#endif
#define
#undef

  • Generowanie błędów i ostrzeżeń kompilacji lub pomijanie wybranych ostrzeżeń kompilatora

    #warning
#error
#line
#pragma [warning | checksum]

  • Nazywanie bloków kodu

    #region
#endregion

UWAGA: W języku C# using System jest dyrektywą, ale nie dyrektywą preprocesora.

Dyrektywy #define, #undef

Dyrektywa preprocesora #define tworzy stałą symboliczną.

#define MYSYMBOL

Tak zdefiniowanej stałej symbolicznej nie przypisujemy wartości. Jedynie deklarujemy istnienie symbolu.

Fakt istnienia symbolu będzie wykorzystany w dyrektywach warunkowych #if, #elif czy atrybucie ConditionalAttribute.

Definiowane stałe symboliczne nie kolidują z nazwami zmiennych programu - należą "do innego świata".

Definiowane symbole umieszczamy zwyczajowo na początku pliku. Wymogiem formalnym jest, aby deklaracja symbolu była przed jego pierwszym użyciem.

Zasięg deklaracji symbolu obowiązuje do końca pliku w którym się on pojawił chyba, że został on "oddefiniowany" dyrektywą #undef.

W środowiskach .NET (.NET Framework, .NET Standard, .NET Core) występują predefiniowane symbole o ustalonym znaczeniu:

  • wykorzystywane w trybie kompilacji

    • DEBUG - pozwala aplikacji na wykorzystanie klasy Debug
    • TRACE - pozwala aplikacji na wykorzystanie klasy Trace

  • informujące o wersji środowiska .NET

Symbole możemy deklarować bezpośrednio w kodzie za pomocą dyrektywy #define lub dostarczać ich deklaracji spoza kodu, wywołując kompilator z odpowiednią opcją:

csc -define:MYSYMBOL -optimize -out:File2.exe *.cs

dotnet msbuild /p:DefineConstants=TRACE;DEBUG;MYSYMBOL /p:Configuration=Debug

ZALECENIE: symbole powinny być definiowane poza kodem, w środowisku zewnętrznym. Wtedy można efektywnie sterować procesem kompilacji.

W środowisku VisualStudio, wywołując formularz właściwości projektu, możemy podać listę definiowanych symboli (obowiązującą dla całego projektu)

ustalając, w jakiej konfiguracji kompilacji definiowane symbole będą obowiązywać.

Domyślnie kompilacja w trybie Debug definiuje symbole DEBUG oraz TRACE. Kompilacja w trybie Release nie definiuje symbolu DEBUG, ale definiuje TRACE. Domyślnie nie definiowany jest w żadnym przypadku symbol RELEASE (podchwytliwe pytania na egzaminie 70-483).

W momencie pojawienia się dyrektywy #undef MYSYMBOL następuje odwołanie definicji tego symbolu.

Kompilacja warunkowa

Dyrektywy warunkowe (#if, #else, #elif, #endif) "konsumują" dyrektywy #define i #undef.

W poniższym przykładzie, jeśli kompilacja odbywa się w trybie Debug (symbol DEBUG jest ustawiony), programista ma dostęp do zmiennej wylosowana za pośrednictwem właściwości GetWylosowana. W kompilacji w trybie Release nie ma do niej dostępu.

    public class Gra
    {  
        private readonly int wylosowana;
    #if DEBUG
        public int GetWylosowana => wylosowana;
    #endif
        // ...
    }

Ponieważ różne wersje środowisk .NET dostarczają różnych interfejsów API, konieczne jest rozdzielanie kompilacji na różne gałęzie.

public class MyClass
{
    static void Main()
    {
#if NET20 || NET35
        HttpClient _client = new HttpClient();
#else
        WebClient _client = new WebClient();
#endif
    //...
    }

    //...
}

W instrukcjach kompilacji warunkowej można używać operatorów ==, !=, && oraz ||. Można również używać nawiasów ( oraz ).

#define MYTEST
using System;
public class MyClass
{
    static void Main()
    {
#if (DEBUG && !MYTEST)
        Console.WriteLine("DEBUG is defined");
#elif (!DEBUG && MYTEST)
        Console.WriteLine("MYTEST is defined");
#elif (DEBUG && MYTEST)
        Console.WriteLine("DEBUG and MYTEST are defined");  
#else
        Console.WriteLine("DEBUG and MYTEST are not defined");
#endif
    }
}

W przestrzeni nazw System.Diagnostics zdefiniowano ConditionalAttribute.

Adnotowanie tym atrybutem klas i metod informuje kompilator, że te fragmenty należy ominąć podczas kompilacji.

//#define OLD
#define NEW

public class Gra
{  
    private readonly int wylosowana;
#if DEBUG
    public int GetWylosowana => wylosowana;
#endif

    [Conditional("NEW"), Conditional("DEBUG")]
    public bool Weryfikuj(int propozycja)
    {
        //...
    }

    [Conditional("OLD")]
    public bool Weryfikuj(int propozycja)
    {
        // nowy kod ...
    }

    // ...
}

[Conditional("DEBUG")]
public class Test { }

Ref: #if DEBUG vs. Conditional("DEBUG") @StackOverflow

W klasie System.Diagnostics.Debugger zdefiniowana jest właściwość IsAttached pozwalająca na stwierdzenie, czy debugger jest dołączony do uruchomionego procesu (może być dołączony zarówno w trybie Debug jak i Release). Często wykorzystuje się testowanie tego faktu w powiązaniu z dyrektywą #if DEBUG, aby ograniczyć obsługę wyjątków podczas kompilacji testów wewnętrznych.

#if DEBUG
    if (System.Diagnostics.Debugger.IsAttached)
    {
        // ...
    }
#endif

Nadawanie nazw blokom kodu - dyrektywy #region i #endregion

Dyrektywa #region pozwala nazwać i określić blok kodu, który można rozwinąć lub zwinąć przy użyciu funkcjonalności tworzenia konspektu (rozwijanie/zwijanie kodu) w edytorze kodu programu (Visual Studio, VS Code). W przypadku długich plików z kodem wygodniejsze jest zwinięcie lub ukrycie jednego lub kilku regionów, aby móc skupić się na tej części pliku, nad którą się obecnie pracuje.

Blok #region musi kończyć się dyrektywą #endregion.

Blok #region nie może nakładać się na blok #if, może natomiast być zagnieżdżony w bloku #if, a blok #if może być zagnieżdżony w bloku #region.

Nazwane bloki kodu stosowane są często w kodzie automatycznie generowanym (np. w WinForm).

Klasy Debug i Trace

Klasa Debug dostarcza metod, które wspierają debugowanie kodu. Metody dołączane są do skompilowanego kodu tylko w trybie Debug (ustawiony symbol DEBUG) - w budowaniu kodu Release nie są kompilowane.

Klasa Trace dostarcza metod, które umożliwiają śledzenie wykonania kodu nie tylko w trybie Debug, ale również w Release (ustawiony symbol TRACE).

Obie klasy dostarczają zestaw tych samych metod - różnice polegają tylko na dołączaniu do kompilacji w odpowiednim trybie.

Z technicznego punktu widzenia metody tych klas opatrzone są adnotacjami kompilacji warunkowej:

  • w klasie Debug:

    [System.Diagnostics.Conditional("DEBUG")]
public static void WriteLine(string message)
{
    TraceInternal.WriteLine(message);
}

  • w klasie Trace:

    [System.Diagnostics.Conditional("TRACE")]
public static void WriteLine(string message)
{
    TraceInternal.WriteLine(message);
}

Obie metody korzystają z tej samej klasy TraceInternal https://referencesource.microsoft.com/#system/compmod/system/diagnostics/TraceInternal.cs.

Do najczęściej używanych metod należą:

  • Assert - sprawdza, czy spełniony jest warunek podany jako parametr. Jeśli nie, zostanie wysłany komunikat o błędzie (nie spełnienie założenia).
  • Write, WriteLine - wypisuje przekazany parametr (wiele przeciążeń)
  • WriteIf, WriteLineIf - wypisuje przekazany parametr, jeśli warunek (również przekazany jako parametr) jest spełniony (wiele przeciążeń).
  • Debug.Print - w zasadzie to samo, co Debug.WriteLine, z tym, że parametrem jest komunikat (typu string), a nie obiekt. Nie występuje w klasie Trace.

Ref: How to: Add Trace Statements to Application Code

Ref: Assertions in Managed Code

Za pomocą obu klas wyniki działania metod domyślnie kierowane są na konsolę debugowania. Miejsce docelowe odbierania wyników działania możesz zmienić, konfigurując Listeners. Miejscem może być konsola, plik, baza danych, ...

Ref: Trace Listeners

// przekierowanie wyników działania metdod klasy `Trace`
// na standardowe wyjście
ConsoleTraceListener myWriter = new ConsoleTraceListener();
Trace.Listeners.Add(myWriter);

// Utworzenie pliku TestFile.txt.
Stream myFile = File.Create("TestFile.txt");

// przekierowanie wyników działania metdod klasy `Trace`
// do pliku
TextWriterTraceListener myTextListener = new TextWriterTraceListener(myFile);
Trace.Listeners.Add(myTextListener);

Kolekcja Listeners jest współużytkowana przez Debug i Trace.

Istnieje możliwość definiowania własnych listenerów, tworząc klasy dziedziczące z TraceListener.
